# Ротация секретов (группа 6)

Процедуры замены чувствительных ключей ArtStudio CMS. Все критические секреты
читаются строго из `config/config.php` или переменных окружения (не из БД) —
см. `crypto`-блок и `SECURITY.md`.

## `encryption_key` (APP_ENCRYPTION_KEY)

Используется для симметричного шифрования чувствительных настроек в БД
(например, токенов соцсетей). Ротация:

1. **Не удаляйте старый ключ сразу** — иначе уже зашифрованные значения станут
   нечитаемыми.
2. Сгенерируйте новый ключ: `php -r 'echo bin2hex(random_bytes(32)), "\n";'`.
3. Временно держите оба ключа (старый — для расшифровки, новый — для записи),
   пере-сохраните зашифрованные настройки через админку (это перешифрует их
   новым ключом), затем уберите старый ключ.
4. Если перешифровать нечего (соцсети/webhooks не настроены) — можно заменить
   ключ сразу.

## Токены соцсетей (Facebook / LinkedIn / Instagram)

Хранятся в настройках раздела `/admin/social` (супер-админ). Ротация: получите
новый токен у провайдера, вставьте в поле токена, сохраните. Старые публикации
(`social_posts`) не требуют перевыпуска. При компрометации — сначала отзовите
токен на стороне провайдера, затем впишите новый.

## Секреты вебхуков (`webhooks.secret`)

HMAC-подпись `X-ArtStudio-Signature` считается на основе секрета вебхука.
Ротация: обновите секрет в `/admin/webhooks` **и** у получателя одновременно
(иначе подпись перестанет сходиться). Доставки в очереди, подписанные старым
секретом, при ретрае будут переподписаны текущим.

## Пароли администраторов / 2FA

- Сброс пароля: `/admin/login` → «Забыли пароль» (токен со сроком 30 мин) или
  CLI `php database/create_admin.php` для нового супер-админа.
- 2FA: сброс/перепривязка TOTP — в профиле; backup-коды регенерируются там же
  (старые аннулируются).
- При компрометации сессии — раздел «Активные сессии» отзывает доступ мгновенно
  (серверный registry).

## SMTP / Telegram

Логин/пароль SMTP и `TELEGRAM_BOT_TOKEN` — только в `config.php`/окружении.
Ротация: замените значение и перезапустите PHP-FPM (или дождитесь следующего
запроса — конфиг читается на старте). Секреты не дублируются в БД.

## Версионирование миграций

Отдельная таблица `schema_migrations` не нужна: применённые миграции
фиксируются в таблице `migrations` (`database/migrate.php`), которая играет ту
же роль — `php database/migrate.php status` показывает применённые и новые.
