# Раздача статики через CDN и инвалидация кэша (задача 127)

Проект отдаёт две категории статики:

- `public/assets/` — CSS/JS/шрифты приложения;
- `public/uploads/public/` — загруженные изображения (и их WebP-варианты).

## Версионирование и cache-busting

URL версионированных ассетов формирует `App\Core\Asset::url()`: к пути
добавляется отпечаток содержимого `?v=<hash>` (из mtime+size файла). При деплое
изменённого файла хэш меняется → URL становится новым ресурсом, и CDN отдаёт
свежую версию **без ручной очистки кэша**. Старые версии можно кэшировать
«навсегда».

Заголовок для этого задаётся в `public/.htaccess`:

```
<FilesMatch "\.(css|js|woff2?|ttf|eot|svg|jpe?g|png|gif|webp|ico)$">
    Header set Cache-Control "public, max-age=31536000, immutable"
</FilesMatch>
```

Загруженные изображения имеют случайные (контентно-уникальные) имена, поэтому
их URL также стабилен и безопасен для «вечного» кэша.

## Рецепт для Cloudflare

1. **DNS/Proxy.** Домен проксируется через Cloudflare (оранжевое облако).
2. **Cache Rules.** Правило: для путей `/assets/*` и `/uploads/public/*` —
   *Cache Eligibility: Eligible for cache*, *Edge TTL: Respect origin* (origin
   уже отдаёт `max-age=31536000, immutable`).
3. **Отдельный CDN-хост (опционально).** Если статика раздаётся с
   `cdn.example.com`, задайте `APP_CDN_URL` и префиксуйте им `Asset::url()`
   (точка расширения; по умолчанию используется относительный путь того же
   домена — CSP `default-src 'self'` не требует изменений).
4. **Инвалидация при деплое.** Не требуется вручную: изменённые файлы получают
   новый `?v=`, а неизменные остаются в кэше. Полная «Purge Everything»
   опциональна для экстренных случаев.

## Проверка

После деплоя новой версии, например, `frontend.css`:

- страница ссылается на `/assets/css/frontend.css?v=<новый hash>`;
- CDN, не найдя такой URL в кэше, запрашивает origin и кэширует новую версию;
- предыдущая версия (`?v=<старый hash>`) остаётся в кэше, но больше не
  запрашивается — конфликтов и «залипшего» кэша нет.
