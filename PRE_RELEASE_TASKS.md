# Задачи перед релизом ArtStudio CMS

Собрано по итогам ревью README.md перед разворачиванием реального сайта.
Задачи сгруппированы по приоритету. Внутри каждой группы — порядок
выполнения рекомендуется соблюдать сверху вниз.

---

## Группа 1 — Безопасность (блокеры, делать первыми)

### 1.1 Прямой доступ к служебным PHP-файлам через веб-сервер

**Проблема:** корневой `.htaccess` (используется в fallback-сценарии, когда
хостинг не позволяет сменить document root на `/public`) блокирует по
блэклисту расширений `.sql|.env|.log` — `.php` в список не попадает.
`config/config.php` и `database/create_admin.php` потенциально доступны
напрямую по URL, в обход `public/index.php` и всей security-логики.
Критично: `create_admin.php` без проверки способа запуска может позволить
создать администратора через браузер, без авторизации.

**Что сделать:**

1. В начало `database/create_admin.php` (и любых других CLI-only скриптов
   в `database/`) добавить:
   ```php
   if (PHP_SAPI !== 'cli') {
       http_response_code(403);
       exit('CLI only');
   }
   ```
2. В корневом `.htaccess` заменить блэклист расширений на deny-all для
   `.php` с учётом старого и нового синтаксиса Apache:
   ```apache
   <IfModule mod_authz_core.c>
       <FilesMatch "\.php$">
           Require all denied
       </FilesMatch>
   </IfModule>
   <IfModule !mod_authz_core.c>
       <FilesMatch "\.php$">
           Order allow,deny
           Deny from all
       </FilesMatch>
   </IfModule>
   ```
3. Убедиться, что это правило не блокирует `public/index.php` и
   `public/download.php` (у `public/` своя `.htaccess` — проверить, что там
   разрешение выставлено явно).
4. Проверить/добавить `public/.htaccess`: `Options -Indexes` (запрет
   листинга директорий) + явное разрешение на `index.php`/`download.php`.
5. Добавить unit-тест в `tests/run.php`: CLI-only проверку вынести в
   тестируемую функцию (например, `App\Core\Cli::assertCli()`), проверить
   что при не-CLI режиме выполнение прерывается с 403.

**Ручная проверка после деплоя (обязательно на реальном сервере, не локально):**
- `https://домен/config/config.php` → 403 или 404
- `https://домен/database/create_admin.php` → 403
- `https://домен/database/schema.sql` → 403 или 404
- `https://домен/storage/logs/` → 403 или 404 (листинг закрыт)

Если хостинг игнорирует `.htaccess` вообще (`AllowOverride None`) — это
находит проблему, которую код не может починить; требовать от хостинга
включить `AllowOverride`, либо переносить сайт.

**Критерий готовности:** все 4 URL выше возвращают 403/404 в проде; тест
на CLI-only проходит.

### 1.2 Согласованный бэкап (БД + файлы одним проходом)

**Проблема:** бэкапы хостинга часто не гарантируют согласованность БД и
файлов (снятие в разное время → рассинхрон ссылок в БД и реальных файлов),
не проверены на восстановление, и являются единой точкой отказа вместе с
самим хостингом.

**Что сделать:**

1. `app/Console/backup_worker.php` (по образцу `mail_worker.php` /
   `social_worker.php`):
   - Снятие дампа БД (`mysqldump` через `proc_open`) + архивация
     `storage/protected_uploads` и `public/uploads/public` **одним
     проходом**.
   - Перед снятием — короткое включение `maintenance_mode` на время
     дампа, либо `FLUSH TABLES WITH READ LOCK`, чтобы исключить рассинхрон
     при параллельной загрузке файла.
   - Контрольная сумма архива (`hash_file('sha256', ...)`) рядом с
     архивом.
   - Ротация: удаление копий старше N дней (значение — из `config.php`).
   - `Logger::event()` — `INFO` при успехе, `CRITICAL` при ошибке
     (используя существующий Telegram-алертинг).
   - Копия архива дублируется во внешнее хранилище (не только локальный
     диск того же сервера) — иначе бэкап бесполезен при полном отказе
     сервера.
2. Cron (пример, раз в сутки):
   ```
   0 3 * * * php /path/to/app/Console/backup_worker.php >> storage/logs/backup_worker.log 2>&1
   ```
3. Скрипт восстановления `app/Console/restore.php`:
   - Принимает путь к архиву, разворачивает БД + файлы в **отдельную**
     тестовую БД/директорию (не боевую), проверяет контрольную сумму,
     выводит отчёт успех/ошибка.
4. **Обязательно один раз прогнать реальное восстановление** на тестовом
   окружении до релиза — бэкап, который ни разу не разворачивали, не
   считается рабочим.

**Критерий готовности:** cron снимает согласованный бэкап ежедневно,
копия есть вне сервера, восстановление один раз проверено вручную и
задокументировано (шаги + сколько заняло по времени).

## Группа 2 — Эксплуатационная надёжность (сильно желательно)

### 2.1 Heartbeat фоновых воркеров

**Проблема:** если cron на хостинге тихо отвалится, узнать об этом можно
только по жалобе пользователя (письма не приходят, вебхуки не летят).

**Что сделать:**
- Каждый воркер (`mail_worker`, `social_worker`, `webhook_worker`,
  `backup_worker`) при старте пишет метку времени в
  `storage/cache/worker_heartbeat_{name}.txt` или в таблицу
  `worker_heartbeats`.
- Расширить существующий `/health`: сверять "сколько времени прошло с
  последнего запуска" каждого воркера по его ожидаемой частоте (например,
  mail_worker должен запускаться раз в 5 минут — если тишина дольше
  15 минут, это `CRITICAL` через `Logger::event()`).

**Критерий готовности:** остановка любого cron-воркера обнаруживается
автоматически и уходит алертом в Telegram в течение разумного времени
(не позже следующего вызова `/health`).

---

### 2.2 Dead-letter статус для очередей после исчерпания ретраев

**Проблема:** ретраи x3 для `webhook_deliveries`/`social_posts` описаны,
но неясно, что происходит после третьей неудачи — записи могут молча
зависать в статусе "в процессе".

**Что сделать:**
- Явный статус `failed` (не просто "не отправлено") после исчерпания
  ретраев.
- В админке (разделы "Вебхуки"/"Соцсети") — фильтр по `failed`.
- Алерт в Telegram при переходе записи в `failed`.

**Критерий готовности:** после 3 неудачных попыток запись видна в
админке как `failed`, а не теряется молча; владелец узнаёт о проблеме
без ручного захода в БД.

## Группа 3 — Управление меню: вложенность (submenu)

**Проблема:** конструктор меню (`/admin/menu`, `menu_items`) сейчас —
плоский список (тип ссылки, язык, сортировка), без вложенности. Для
корпоративного сайта нередко нужен dropdown ("Услуги" → под-пункты).

**Что сделать:**

1. **Схема:** добавить в `menu_items` поле `parent_id` (nullable, FK на
   `menu_items.id`, `ON DELETE CASCADE`) через миграцию в
   `database/migrations/` (не менять `schema.sql` напрямую).
2. **Ограничение глубины и защита от циклов:** на сервере жёстко
   ограничить вложенность одним уровнем — если у выбранного `parent_id`
   уже есть свой `parent_id`, отклонять с понятной ошибкой. Отдельно
   проверять, что `parent_id` не равен `id` самого пункта.
3. **Админка:** переиспользовать существующий HTML5 Drag-and-Drop (по
   образцу задачи 134 — сортировка блоков), добавить перетаскивание
   "внутрь" другого пункта (меняет `parent_id`), визуальный отступ для
   дочерних пунктов в списке. AJAX-сохранение `sort_order` + `parent_id`
   одним запросом (по образцу `Block::reorder`).
4. **Фронтенд:** в `app/Views/site/_header.php` группировать пункты по
   `parent_id`; dropdown на desktop — hover + focus (использовать уже
   имеющуюся работу над `:focus-visible`); на мобильных — по тапу, с
   индикатором (шеврон) у пунктов с детьми. URL и языковой префикс —
   без изменений в существующей логике генерации ссылок.
5. **Обратная совместимость:** для существующих установок миграция
   добавляет `parent_id = NULL` по умолчанию — все текущие пункты
   остаются на верхнем уровне без ручных действий.

**Критерий готовности:** можно создать пункт меню и вложить под него
2-3 дочерних через drag-and-drop; глубина ограничена одним уровнем;
dropdown корректно работает на desktop и мобильных на всех 3 языках;
удаление родителя каскадно удаляет детей (с явным предупреждением в UI
перед удалением); тест на цикл/превышение глубины в `tests/run.php`.

## Группа 4 — Библиотека блоков конструктора страниц

Текущий набор: `text, html, cta, advantages, slider, gallery, form`.
Приоритет добавления новых типов (по частоте запроса у корпоративных
клиентов):

1. **Testimonials/отзывы** — цитата + имя + компания + фото, карусель.
2. **Counters/статистика** — анимированный счётчик чисел ("500+ клиентов").
   Переиспользовать существующий `IntersectionObserver` из `frontend.js`
   (уже используется для `data-reveal`) — вместо fade-in анимировать
   инкремент числа.
3. **Columns/сетка с вложенными блоками** (см. детали в 4.1 ниже) —
   самая трудоёмкая, но самая универсальная задача.
4. **Блоки "Список команды" / "Список проектов"** — не новый тип с нуля,
   а блок-обёртка, выводящая данные уже существующих сущностей (`/admin/team`,
   `/admin/projects`) на произвольной странице (например, "3 последних
   проекта" на главной).

Логотипы партнёров и FAQ/аккордеон можно закрыть позже: логотипы —
переиспользованием существующего `slider`-блока с другим шаблоном, FAQ —
отдельная небольшая задача на следующую итерацию.

### 4.1 Блок "Columns" с вложенными блоками

**Архитектурное решение** (сейчас блоки — плоский список
`blocks(page_id, sort_order)`, вложенности нет):

1. Добавить в `blocks` поля `parent_block_id` (nullable, FK на `blocks.id`)
   и `column_index` (int).
2. Блок типа `columns` в `data` хранит только структуру: количество
   колонок (2/3/4), ширины, gap. Содержимое — обычные дочерние блоки с
   `parent_block_id`/`column_index`.
3. `BlockRenderer::render()`: для блока `columns` — получить дочерние
   блоки, сгруппировать по `column_index`, рекурсивно вызвать `render()`
   для каждого (переиспользовать существующий метод).
4. `CssScoper` не требует изменений — скоуп идёт по `id` конкретного
   блока независимо от вложенности.
5. В админке: в каждой колонке — кнопка "Добавить блок" с тем же
   селектором типов, что и на верхнем уровне страницы, передавая
   `parent_block_id`/`column_index`.
6. Запретить вкладывать `columns` в `columns` (проверка на сервере при
   сохранении).

**Критерий готовности:** блок "3 колонки" с разными типами вложенных
блоков (иконка+текст, картинка, CTA) рендерится корректно, каждый
вложенный блок сохраняет независимые scoped-стили.

### 4.2 Тип анимации появления (не только вкл/выкл)

1. Заменить `blocks.data._reveal: bool` на
   `_reveal: {enabled: bool, type: string}` (`fade`, `slide-up`,
   `slide-left`, `slide-right`, `zoom-in`). Обратная совместимость: старое
   булево `true` трактовать как `{enabled: true, type: 'fade'}` через
   мердж с дефолтами (как уже сделано для остальных полей блока).
2. Разметка: `data-reveal="true" data-reveal-type="slide-up"`.
3. `frontend.js`: `IntersectionObserver` добавляет `reveal-active` и
   `reveal-{type}`; CSS-переходы — в общем frontend CSS.
4. `prefers-reduced-motion` по-прежнему отключает анимацию полностью,
   независимо от типа.
5. UI: select с типами анимации вместо чекбокса.

### 4.3 SVG-иконки — код или файл

**(а) Через код:**
- Поле `icon_svg` в форме блока (аналогично `custom_css`).
- Обязательно прогонять через существующий
  `App\Core\Uploader::sanitizeSvgString()` при сохранении — хранить уже
  очищенную версию, не сырой ввод.

**(б) Через файл:**
- Переиспользовать `FileController::library` + `data-media-pick`,
  добавить фильтр по `image/svg+xml`.
- Хранить `icon_file_id`, при рендере — инлайнить содержимое (файл уже
  прошёл `sanitizeSvgString` при аплоаде, повторная санитизация не
  требуется) либо использовать `<img>` (проще, но не даёт красить через
  `currentColor`).

**UI:** переключатель "Код" / "Файл из библиотеки" — не два активных
поля одновременно.

**Критерий готовности (4.2 + 4.3):** для блока с иконкой можно выбрать
тип анимации из 4-5 вариантов и способ добавления иконки (код или файл);
попытка вставить SVG с `<script>` внутри — скрипт вырезан после
сохранения.

## Группа 5 — Управление дизайном: дополнительно

### 5.1 История версий блока (мастхэв)

**Что сделать:**
- Таблица `block_revisions` (`block_id`, `data`, `custom_css`,
  `created_at`, `created_by`).
- При каждом сохранении блока — класть текущее состояние в
  `block_revisions` перед перезаписью.
- Хранить последние 20 ревизий на блок (старше — удалять).
- В UI — кнопка "История изменений" со списком (дата, автор) и
  восстановлением; восстановление тоже создаёт новую ревизию.

**Критерий готовности:** можно откатить блок к любой из последних 20
версий через UI без прямого доступа к БД.

### 5.2 Предпросмотр страницы до публикации (мастхэв)

**Что сделать:**
- `GET /admin/pages/{id}/preview` (и аналогично для новостей) — рендер
  через `BlockRenderer::renderPage()`, но с несохранёнными/черновыми
  данными, доступ только авторизованным (`Auth::check()`), `noindex`,
  не в sitemap/кэше.
- Кнопка "Предпросмотр" на странице редактирования.

**Критерий готовности:** редактор видит страницу целиком со всеми
блоками и scoped CSS до публикации, включая ещё не сохранённые как
основные правки.

## Группа 6 — Не блокирует релиз, отложить в бэклог

- Глобальный custom CSS/JS сайта вне блоков
- Расширение цветовой палитры настроек дизайна (сейчас неясно, сколько
  переменных доступно — уточнить у бизнеса перед решением)
- Дополнительные типы лейаута страницы (например, лендинг без шапки/футера)
- Полноценный визуальный drag-and-drop дизайнер (нативная сортировка
  блоков уже достаточна для MVP)
- FAQ/аккордеон, логотипы партнёров (как отдельный тип, не через `slider`)
- GDPR-очистка старых заявок форм как автоматизированный cron
  (`gdpr_cleanup.php`) — подтвердить, что срок хранения ПДн в настройках
  реально к чему-то привязан кодом
- Ротация секретов (`encryption_key`, токены соцсетей/вебхуков) —
  задокументировать процедуру
- Таблица `schema_migrations` с версионированием применённых миграций
  (сейчас применяются вручную по списку в README)
- File-lock (`flock()`) на cron-воркерах — защита от повторного запуска
  поверх незавершённого предыдущего
- Нагрузочное тестирование очередей (`webhook_deliveries`, `social_posts`)
- Внешний security-аудит/пентест — оправдан при росте публичности проекта
  или появлении платежей/большого объёма персональных данных

## Рекомендуемый порядок выполнения перед реальным релизом

1. Группа 1 (1.1 → 1.2) — безопасность, без вариантов.
2. Группа 3 — вложенность меню (относительно небольшая, высокая видимая
   ценность для клиента).
3. Группа 5 (5.1 история версий, 5.2 предпросмотр) — дёшево, сильно
   снижает риск "сломанного дизайна" в проде.
4. Группа 4 — блоки (начать с 4.1 Columns, затем testimonials/counters,
   затем 4.2/4.3 как сквозные улучшения).
5. Группа 2 — heartbeat и dead-letter, желательно до релиза, но можно
   в первую неделю после, если сроки поджимают.
6. Группа 6 — по мере реального спроса от клиента/редакторов, не раньше.
